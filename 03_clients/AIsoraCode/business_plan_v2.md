## 4. 設定一覧（やるべきこと / やってはいけないこと）

### 絶対にやる設定（ガードレール）

| # | 設定項目 | 内容 | 理由 |
|---|---|---|---|
| 1 | .envファイルの保護 | ignorePaths に .env を追加 | 機密情報の漏洩防止 |
| 2 | 機密ファイルの除外 | *.key, *.pem, credentials.json 等を除外 | APIキー・証明書の保護 |
| 3 | CLAUDE.mdにセキュリティルール記載 | 「.envを読むな」「機密情報を出力するな」 | AI側の行動制限 |
| 4 | settings.jsonの権限設定 | Deny/Ask/Allowの3段階設定 | 誤操作防止 |
| 5 | git commit前チェックルール | 機密情報がコミットに含まれないか確認 | 流出防止 |
| 6 | 破壊的操作の制限 | rm -rf, git reset --hard等に確認必須 | データ消失防止 |

### 絶対にやってはいけないこと（クライアントに必ず伝える）

| # | 禁止事項 | 理由 |
|---|---|---|
| 1 | APIキー・パスワードをCLAUDE.mdに書く | AI経由で漏洩するリスク |
| 2 | .envファイルをgitにコミットする | GitHubに機密情報が公開される |
| 3 | 全自動モード（auto-accept）で放置 | AIが意図しない操作を実行するリスク |
| 4 | Claude Codeに本番DBへの直接アクセスを許可 | データ破壊リスク |
| 5 | 外部URLへの未検証アクセスを許可 | プロンプトインジェクション攻撃のリスク |
| 6 | 顧客の個人情報をClaude Codeに渡す | プライバシー法違反のリスク |

### ルール化すべき運用ルール（CLAUDE.mdに組み込む）

| # | ルール | 内容 |
|---|---|---|
| 1 | ファイル変更は承認制 | 重要ファイルの変更前に確認を求める |
| 2 | 外部送信は禁止 | 未承認の外部URLへのデータ送信を禁止 |
| 3 | コーディング規約の遵守 | 会社のコーディングスタイルに従う |
| 4 | ログ記録 | 重要な作業はログとして記録する |
| 5 | エスカレーションルール | 判断に迷ったら人間に確認する |

## 5. 事業売却への道筋

### なぜ売却できるのか

この事業が売却可能な資産になる条件:

| 条件 | 状態 | 構築方法 |
|---|---|---|
| 属人性がない | パートナー制度 + マニュアル化 | 型化 → パートナー研修 → 品質管理 |
| 再現可能な収益 | 月40件 × 25万の安定受注 | SNS集客の仕組み化 + 紹介ネットワーク |
| 知的財産がある | テンプレート・スキルパック・マニュアル | コンサルしながらデータを蓄積 |
| 成長性がある | Claude Code市場の拡大 | 市場の波に乗っている |

### 売却シナリオ

| 指標 | 数値 |
|---|---|
| 年間売上（月1,000万達成後） | 1.2億円 |
| 年間利益率 | 50%（パートナーマージン後） |
| 年間利益 | 6,000万円 |
| 売却マルチプル | 3〜5x（コンサル事業の相場） |
| **売却想定額** | **1.8億〜3億円** |
| 2人で分配 | **9,000万〜1.5億円/人** |

### 売却までのマイルストーン

| フェーズ | 期間 | 目標 |
|---|---|---|
| Phase 1: 型作り | Month 1-3 | 自分たちで10件こなし、テンプレートと手順を確立 |
| Phase 2: パートナー展開 | Month 4-6 | パートナー5人を育成、月20件体制 |
| Phase 3: スケール | Month 7-12 | パートナー10人+、月40件、月商1,000万達成 |
| Phase 4: 安定化 | Year 2 | 月商1,000万を6ヶ月以上維持。セルフサービスキット開発 |
| Phase 5: 売却準備 | Year 2後半 | マニュアル整備、財務整理、買い手探し |
| Phase 6: 売却 | Year 3 | **EXIT** |
